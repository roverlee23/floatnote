# FloatNote 窗口与全屏空间策略

这份文档定义 FloatNote 在 macOS 桌面、全屏空间和图钉状态下的统一行为。以后修改窗口代码时，应以本文件为准，避免为单一现象增加新的例外分支。

## 1. 产品目标

FloatNote 是轻量便签，但用户需要在任何界面快速唤起并编辑。因此窗口必须同时满足：

- 在普通桌面上，它是正常 App：会被别的 App 盖住，也只停留在当前桌面。
- 在真正的全屏 App 上，可以临时调出便签进行记录。
- 只有用户主动开启图钉时，便签才会跨桌面、跨全屏空间持续浮动。
- 目标是：快捷键和 Dock 图标都能在**当前所在空间**切换 FloatNote；顶部状态栏图标只负责拉起到前面。

## 2. 唯一的三种窗口模式

| 模式 | 进入条件 | 层级与空间范围 | 退出方式 |
| --- | --- | --- | --- |
| `Desktop` | 未图钉，当前是普通桌面 | 普通窗口层级；显示时进入当前桌面；不跨空间 | 用户切换 App、手动收起或关闭 |
| `Fullscreen` | 未图钉，当前是 macOS 真全屏空间 | 只临时附着在当前全屏空间上方 | 点击全屏 App 其他区域、切到任意其他空间、关闭 |
| `Pinned` | 用户开启图钉 | 置顶，并允许跨所有桌面与全屏空间 | 关闭图钉或关闭窗口 |

图钉是唯一允许跨空间的开关。任何未图钉窗口都不得因为历史状态而跟到别的空间。

## 3. 三个入口的产品语义相近，但系统事件不同

用户角度看，快捷键、Dock 和顶部状态栏都是“把 FloatNote 调出来”；但 macOS 底层处理不同，不能只因为产品语义相同就强行共用同一个原生处理函数。

| 入口 | 系统先做什么 | 正确的产品行为 | 实施要点 |
| --- | --- | --- | --- |
| 全局快捷键 | 不激活整个 App，只把事件交给 FloatNote | 隐藏/在后面时调起；已在前面时收起 | 可以直接读取当前空间并 `show + orderFront + makeKey` |
| 顶部状态栏图标 | FloatNote 自己收到状态栏点击，不会先切换桌面 | 只调起到前面，不收起正在编辑的便签 | 保持单独的 `show_and_focus_window` 路径 |
| Dock 图标 | macOS 会激活整个 App，并可能跳到它原本所属桌面 | 目标是像快捷键一样在当前全屏调起 | 必须在系统默认跳桌面前处理“Dock 桥接”，不能在跳完后补救 |

因此，“统一”应只统一到产品层的最终结果，不能统一到原生事件的第一步。快捷键和状态栏已经稳定；Dock 是单独的 macOS 兼容问题。

## 4. 如何判断真全屏

不能通过“某个窗口是否刚好铺满屏幕”判断全屏。普通桌面 App 最大化后也会铺满屏幕，旧实现因此将普通桌面误判为全屏，造成窗口跑回旧全屏空间。

实现上读取 macOS Mission Control 的当前空间类型：

- `type = 4`：真全屏空间，使用 `Fullscreen`；
- 其他类型：普通桌面，使用 `Desktop`。

这项判断只在“唤起窗口”或“切换图钉”时执行，不在每次失焦时重新猜测当前环境。

## 5. 各模式的原生窗口设置

| 模式 | macOS collection behavior | 窗口层级 |
| --- | --- | --- |
| `Desktop` | `MoveToActiveSpace + FullScreenNone` | 普通层级，不是浮动面板 |
| `Fullscreen` | `MoveToActiveSpace + FullScreenAuxiliary` | 当前全屏 App 上方 |
| `Pinned` | `CanJoinAllSpaces + FullScreenAuxiliary` | 始终置顶、跨空间 |

不要把 `CanJoinAllSpaces` 用在未图钉模式；不要在 `apply_window_mode` 内抢焦点。模式函数只负责设置层级和空间归属，只有真正的唤起操作才负责 `show + orderFront + makeKey`。

### Dock 桥接层（后续修 Dock 的唯一正确切入点）

第一版之所以能在全屏内点击 Dock 后正常出现，是因为窗口始终使用 `CanJoinAllSpaces + FullScreenAuxiliary`：它在 Dock 点击前就已经有资格出现在所有全屏空间里。代价是未图钉时也像跨空间悬浮窗口，无法满足后续“桌面上是普通 App”的要求。

后续正确方案不能回到“全局跨空间”，而应增加一个只为 Dock 服务的桥接层：

1. Dock 点击进入 `applicationShouldHandleReopen` 时，先读取并保存当前空间编号；
2. 在 macOS 默认激活/跳桌面动作之前，把隐藏窗口临时设为“可加入当前全屏”的辅助窗口；
3. 返回“本 App 已处理 Dock 点击”，阻止系统把旧桌面窗口当作默认恢复目标；
4. 确认窗口仍在保存的空间编号后，再显示并置前；
5. 成功显示后恢复未图钉的 `Fullscreen` 或 `Desktop` 行为，绝不把 `CanJoinAllSpaces` 留给未图钉窗口。

这条链路必须以真实 Dock 点击做人工回归；只测快捷键、只看 `Reopen` 日志都不足以证明 Dock 成功。

## 6. 退后台与状态清理

未图钉 `Fullscreen` 模式下：

- 用户点击全屏 App 的其他区域：失焦，退后台；
- 用户切换到任意其他空间：退后台；
- 用户点击关闭：退后台。

空间变化通知本身不代表用户真的切了空间。进入或调整全屏辅助窗口时，macOS 也可能发送一次通知；因此程序保存唤起时的全屏空间编号，只在编号与当前空间不同的时候退后台。

退后台统一调用 `dismiss_window`：逻辑状态必须重置，避免隐藏窗口带着旧全屏状态影响下一次快捷键唤起。清理原生窗口空间属性时要额外保留 Dock 桥接的测试；不能为了清理方便而重新让 Dock 只能找到旧桌面窗口。

未图钉 `Desktop` 模式下，失焦不隐藏；它只按普通 App 的层级关系落在其他 App 后面。

## 7. 修改时必须遵守的边界

- 入口可以有不同的原生适配层，但最后必须落到明确的产品动作：`toggle_window` 或 `show_and_focus_window`。
- 新的“收起”路径必须调用 `dismiss_window`，不能只调用 `hide()`。
- 不要以窗口尺寸、所有者名称或截图结果判断全屏。
- 图钉状态变化后立刻重新应用窗口模式。
- 日志不得记录便签正文，只记录入口、模式、空间类型、可见和焦点状态。
- 不要因为 Dock 有问题而改动快捷键、状态栏、表格或图钉逻辑；Dock 只能在 Dock 桥接层内修。

## 8. 每次改动后的人工验证清单

1. 普通桌面：快捷键和 Dock 均能在当前桌面调起、再点一次收起；状态栏图标只调起、不收起。
2. 普通桌面：FloatNote 在前时，切换到另一 App 后应落到后面，不应消失或置顶。
3. 全屏：快捷键和 Dock 都应在当前全屏上调起；状态栏图标应在当前全屏只调起；点击全屏 App 外部区域后应退后台。
4. 全屏：未图钉时左右切换到其他空间，FloatNote 应退后台；返回原全屏也不应自动出现。
5. 图钉：从桌面切到全屏、从全屏切到桌面，FloatNote 都应持续存在。
6. 图钉关闭后：马上恢复对应当前空间的普通规则，不能残留跨空间行为。

若问题复现，提供 `~/Library/Application Support/floating-notes/window-debug.log`。先按日志确认入口、当前空间类型和最终模式，再改代码；不要直接追加新的特殊判断。

## 9. 本次复盘：为什么第一版成功、优化后又失效

第一版采用的是“窗口始终跨空间”的简单方案。它在 Dock、快捷键和状态栏下都很稳定，因为窗口从来没有失去全屏空间资格；但它不符合“未图钉时桌面要像普通 App”的新要求。

后续优化把未图钉窗口改为只属于当前桌面，并补上了全屏临时模式、切屏退后台、空间编号判断。这些规则本身解决了快捷键和状态栏场景，却让 Dock 暴露出系统级差异：Dock 不是普通点击，而是 App 重新激活。若在 Dock 默认动作发生后再显示窗口，macOS 已经可能把 App 带回桌面，后续代码只能看到“已经切屏”的结果。

经验结论：先写清楚“空间资格、窗口层级、产品状态、入口事件”四件事，再改任何一件。尤其在 macOS 上，不能把 Dock、状态栏和快捷键当作同一种底层事件。第一版的可用行为应保留为回归基线；优化必须在不破坏该基线的前提下推进。
